# Cherry Studio 工具调用详解

本文档集中说明 Cherry Studio 中「模型如何发起工具调用、谁来执行、结果如何回到对话」的完整链路，并与 `packages/aiCore`、渲染进程 Legacy 中间件、主进程 MCP 的职责划分对齐。

## 1. 概念分层

工具调用在仓库里至少涉及三层，不要混为一谈：

| 层次 | 职责 | 典型位置 |
| --- | --- | --- |
| **协议与模型侧** | 模型输出「要调哪个工具、参数是什么」（原生 Function Call 或 XML 文本） | 各 Provider / Vercel AI SDK 流事件 |
| **产品编排（渲染进程）** | 把流变成 Chunk、权限确认、调 IPC、递归续聊、更新消息块 UI | `src/renderer/src/aiCore/`、`StreamProcessingService`、`BlockManager` |
| **执行环境（主进程）** | 连接 MCP Server、真正执行 `callTool`、返回结构化结果 | `MCPService`、`IpcChannel.Mcp_CallTool` |

用户看到的「工具卡片 / 等待确认 / 执行中 / 完成」由 **Chunk + MessageBlock** 驱动；真正跑 MCP 的是 **主进程**。

---

## 2. 工具类型（产品语义）

应用内将工具归为三类（与 UI 与路由逻辑一致）：

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| `mcp` | MCP 服务器注册的工具 | 文件、数据库、第三方 API 等 |
| `builtin` | 应用内置、在渲染侧即可完成的工具 | 如 `think`（思考工具） |
| `provider` | 由模型厂商提供的原生能力 | 如部分场景下的 `web_search` |

MCP 类工具最终通过 `window.api.mcp.callTool` → 主进程 `MCPService.callTool` 执行；内置类可在渲染进程直接处理。

---

## 3. 两种「模型侧」形态

### 3.1 原生 Function Calling（AI SDK / Provider）

模型通过 API 的 `tools` 与流式事件输出结构化 tool call（例如 tool name、arguments、tool call id）。渲染侧通过 **AI SDK 流适配**（如 `AiSdkToChunkAdapter`）把 `tool-input-*` 等事件转成 **`MCP_TOOL_*` 系列 Chunk**，再进入权限与执行逻辑。

适用于已声明支持工具调用的模型与 Provider。

### 3.2 Prompt-based（XML 标签）

对**不支持原生 Function Call** 的模型，在系统提示中注入工具定义，约定模型用 XML 输出调用意图，例如：

```xml
<tool_use>
  <name>{tool_name}</name>
  <arguments>{json_arguments}</arguments>
</tool_use>
```

解析由 **`ToolUseExtractionMiddleware`**（Legacy 管道）与 **`packages/aiCore` 内置 `promptToolUsePlugin`**（`promptToolUsePlugin.ts` + `TagExtractor`）等路径配合完成；解析结果同样会进入后续的「待执行工具」管线。

详见插件目录：`packages/aiCore/src/core/plugins/built-in/toolUsePlugin/`。

### 3.3 与 Google 等厂商原生工具的关系

部分 Google 场景下，厂商侧工具通过 **`googleToolsPlugin`** 合并进 AI SDK 的 `ToolSet`，与通用 MCP 工具并存，由同一套执行与流式抽象消费。

## 4. Legacy 中间件管道中与工具相关的环节

渲染进程 Legacy 链路的注册顺序见 [请求流程完整指南 · 5.2](./请求流程完整指南.md#52-中间件注册表)。与工具直接相关的中间件包括：

| 中间件 | 作用 |
| --- | --- |
| **McpToolChunkMiddleware** | 在流上挂接工具处理：检测/消费工具调用、触发递归续聊、控制最大递归深度等 |
| **ToolUseExtractionMiddleware** | 从文本流中提取 Prompt-based 工具调用 |
| **WebSearchMiddleware** | 网页搜索类能力（与 MCP 工具并列的另一种「外部能力」chunk 路径） |

注意：`builder.ts` 示例与 `register.ts` 表格中的顺序可能用于说明不同组合；以 **`register.ts` 实际注册顺序** 为准排查问题。

## 5. Chunk 类型与流式回调

工具生命周期在流中体现为一组 **ChunkType**（定义见 `src/renderer/src/types/chunk.ts`），主要包括：

- `MCP_TOOL_CREATED`：已识别工具调用，创建 `ToolCallInfo` 等上下文
- `MCP_TOOL_PENDING`：等待用户确认或自动批准策略判定
- `MCP_TOOL_IN_PROGRESS`：已通过 IPC 发起执行
- `MCP_TOOL_STREAMING`：工具输出流式增量（若适用）
- `MCP_TOOL_COMPLETE`：执行结束，结果写回对话上下文

`StreamProcessingService` 将 `MCP_TOOL_PENDING` 等映射到 **`onToolCallPending` / `onToolCallComplete`** 等回调，再由消息块与 Redux 更新 UI。

界面侧 **`MessageBlockType.TOOL`** 对应工具块（如 `ToolBlock.tsx`），与文本块、思考块等并列。

## 6. 端到端执行流程（摘要）

下列步骤与 [请求流程完整指南 §7.3](./请求流程完整指南.md#73-工具调用完整流程) 大图一致，此处用文字串起来便于检索：

1. **检测**：`AiSdkToChunkAdapter`（原生流）或 `ToolUseExtractionMiddleware`（Prompt 流）识别工具调用。
2. **建块**：发出 `MCP_TOOL_CREATED` / `MCP_TOOL_PENDING`，展示等待或自动批准。
3. **权限**：`toolPermissions` 判断是否自动批准；否则弹窗确认。
4. **执行**：`MCP_TOOL_IN_PROGRESS` 期间通过 **`window.api.mcp.callTool`** 调用主进程 **`MCPService.callTool`**；内置工具在渲染进程直接处理。
5. **完成**：`MCP_TOOL_COMPLETE`，必要时对知识库引用等做特殊处理。
6. **续聊**：把工具结果注入上下文，**再次请求模型**；递归深度有上限（文档中写明为 **20**），防止死循环。

主进程 MCP 支持多种传输：**Stdio**、**SSE**、**Streamable HTTP** 等（见 `MCPService`）。

---

## 7. 权限与状态

- **自动批准**：可按工具/服务器配置，跳过人工确认。
- **人工确认**：`MCP_TOOL_PENDING` 阶段阻塞直至用户允许或拒绝。
- **简单状态机**：`pending → invoking → done`，或中途 `cancelled`（见指南 §7.7）。

持久化注意：Redux 的 `toolPermissions` 等可能与 `redux-persist` 策略相关，调试时留意是否从缓存恢复旧状态。

---

## 8. 超时与错误

常规请求超时配置在指南其它章节有汇总表；**MCP 工具调用** 往往单独配置（文档中常见为 **默认 60 秒** 量级，以设置项为准）。超时或 MCP 错误应通过 **ERROR Chunk** 或块状态 **`ERROR`** 呈现，并走统一错误中间件。

## 10. 新架构（`index_new` / `packages/aiCore`）与 Legacy 并存

当前仓库处于迁移期：**Legacy**（`src/renderer/src/aiCore/legacy`）仍是大量会话路径的实现所在；**`packages/aiCore`** 提供统一的 Provider、中间件与内置插件（含 Prompt 工具）。阅读代码时请先确认调用栈走的是哪条入口（例如 `legacy/index.ts` 与 `index_new.ts`），避免「文档与实现路径不一致」的困惑。

若仅关心 **Claude Code Agent** 子系统，其 tool-call 状态机与会话 ID 命名空间见 [claude-agent-sdk-design.md](./claude-agent-sdk-design.md)，与上述「普通聊天 + MCP」路径不完全相同。
