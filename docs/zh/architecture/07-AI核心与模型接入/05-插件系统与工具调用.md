# 05-插件系统与工具调用

插件系统是 Cherry Studio AI 链路的扩展主轴。当前实现是"aiCore 插件引擎 + 渲染层插件编排"双层模型。

## 双层架构

```
┌───────────────────────────────────────────────────────────┐
│  渲染层 PluginBuilder（策略装配）                            │
│  根据 provider / model / config 条件决定启用哪些插件          │
│  src/renderer/src/aiCore/plugins/PluginBuilder.ts          │
├───────────────────────────────────────────────────────────┤
│  aiCore PluginEngine（生命周期执行）                         │
│  packages/aiCore/src/core/plugins/                         │
│  ├── PluginManager    插件排序与钩子执行策略                  │
│  ├── built-in/        内置插件实现                           │
│  │   ├── providerToolPlugin.ts   provider 原生工具注入       │
│  │   ├── webSearchPlugin/        模型原生 web search 配置    │
│  │   └── toolUsePlugin/          Prompt 模拟工具调用          │
│  │       ├── promptToolUsePlugin.ts  主插件                 │
│  │       ├── ToolExecutor.ts         工具执行器              │
│  │       ├── StreamEventManager.ts   流事件管理              │
│  │       ├── tagExtraction.ts        流式 XML 标签提取器      │
│  │       └── type.ts                类型定义                 │
│  └── types.ts         AiPlugin 接口定义                     │
└───────────────────────────────────────────────────────────┘
```

## aiCore 插件接口（AiPlugin）

`AiPlugin<TParams, TResult>` 定义了 8 个钩子，按执行策略分为四类：

### First 钩子（首个有效结果）

| 钩子 | 职责 |
|------|------|
| `resolveModel` | 模型解析。第一个返回非 null 模型的插件胜出，用于模型 ID 解析的 provider 级差异 |
| `loadTemplate` | 模板加载。加载提示词模板，第一个返回非 null 模板的插件胜出 |

### Sequential 钩子（链式执行）

| 钩子 | 职责 | 执行方式 |
|------|------|---------|
| `configureContext` | 配置上下文。每个插件依次执行，用于向 `AiRequestContext` 注入元数据 | 顺序调用 |
| `transformParams` | 参数转换。每个插件返回 `Partial<TParams>`，合并到累积参数中 | 链式合并 |
| `transformResult` | 结果转换。每个插件接收完整结果并可选择性修改 | 链式传递 |

### Parallel 钩子（并行执行）

| 钩子 | 职责 | 执行方式 |
|------|------|---------|
| `onRequestStart` | 请求开始。日志记录、span 创建等 | `Promise.all()` 并行 |
| `onRequestEnd` | 请求结束。记忆写回、token 统计等 | `Promise.all()` 并行 |
| `onError` | 错误处理。错误日志、span 标记等 | `Promise.all()` 并行 |

### Stream 钩子（流处理）

| 钩子 | 职责 |
|------|------|
| `transformStream` | 返回 `TransformStream<TextStreamPart, TextStreamPart>`，直接接入 AI SDK 的 `transform` 选项 |

## PluginManager 执行策略

`PluginManager` 负责插件排序和钩子调度：

1. **排序**：按 `enforce` 字段分为 `pre -> normal -> post` 三组
2. **First 执行**：遍历插件，返回第一个非 null 结果
3. **transformParams 链式合并**：依次调用，每个插件返回 `Partial<TParams>`，合并到 `{ ...result, ...partial }`
4. **transformResult 链式传递**：依次调用，每个插件接收完整结果并返回修改后的结果
5. **configureContext 顺序调用**：每个插件依次配置上下文
6. **Parallel 执行**：`Promise.all()` 并行执行所有插件的对应钩子
7. **Stream 收集**：收集所有 `transformStream` 返回的 `TransformStream`，传入 AI SDK

## 插件生命周期在 PluginEngine 中的完整流程

`PluginEngine` 是 aiCore 内部的插件生命周期管理器。一次请求的完整插件流程如下：

### 非流式请求（executeWithPlugins）

```
1. configureContext      → 所有插件依次配置上下文
2. onRequestStart        → 并行执行所有插件的 onStart 钩子
3. resolveModel          → First 策略，第一个返回模型的插件胜出
4. transformParams       → 链式合并，每个插件返回 Partial<TParams>
5. transformStream       → 收集所有 TransformStream
6. [执行 AI SDK generateText]
7. transformResult       → 链式传递，依次转换结果
8. onRequestEnd          → 并行执行所有插件的 onEnd 钩子
```

### 流式请求（executeStreamWithPlugins）

```
1. configureContext      → 所有插件依次配置上下文
2. onRequestStart        → 并行执行所有插件的 onStart 钩子
3. resolveModel          → First 策略
4. transformParams       → 链式合并
5. transformStream       → 收集所有 TransformStream，串联到 AI SDK 流
6. [执行 AI SDK streamText]
7. 流式执行期间：
   - transformStream 的 TransformStream 逐块处理流数据
   - PromptToolUsePlugin 可在流中检测工具调用并递归执行
8. onRequestEnd          → 流结束后并行执行 onEnd 钩子
```

### 递归控制

插件执行支持递归调用（如 `PromptToolUsePlugin` 在检测到工具调用后需要重新发起请求），默认最大递归深度为 10，防止栈溢出。

## PluginBuilder 的编排策略

`buildPlugins()` 位于 `src/renderer/src/aiCore/plugins/PluginBuilder.ts`，根据 provider / model / config 条件组装插件。以下是完整的装配逻辑：

### 装配决策树

```
buildPlugins({ provider, model, config })
  │
  ├─ [条件] topicId && developerMode
  │   └─ createTelemetryPlugin(topicId, assistant)     // OpenTelemetry 追踪
  │
  ├─ [始终] createPdfCompatibilityPlugin(provider, model)  // PDF FilePart → TextPart
  │
  ├─ [条件] provider === openai || azure-openai
  │   └─ createReasoningExtractionPlugin(tagName)     // 提取 <thinking>/<think> 标签
  │
  ├─ [条件] !config.streamOutput
  │   └─ createSimulateStreamingPlugin()              // 非流式 → 模拟流式
  │
  ├─ [条件] provider.anthropicCacheControl.tokenThreshold
  │   └─ createAnthropicCachePlugin(provider)         // Anthropic 缓存控制
  │
  ├─ [条件] provider === openrouter
  │   └─ createOpenrouterReasoningPlugin()            // 清洗 [REDACTED] 推理块
  │
  ├─ [条件] provider === ovms && mcpTools.length > 0
  │   └─ createNoThinkPlugin()                        // OVMS + MCP 禁用思考
  │
  ├─ [条件] Qwen3 模型 && !支持 enable_thinking 的 provider
  │   └─ createQwenThinkingPlugin(enableThinking)     // /think 或 /no_think 后缀
  │
  ├─ [条件] isGemini3Model(model)
  │   └─ createSkipGeminiThoughtSignaturePlugin()     // 跳过 Gemini 思考签名干扰
  │
  ├─ [条件] enableWebSearch && webSearchPluginConfig
  │   └─ providerToolPlugin('webSearch', config)      // 模型原生 web search 注入
  │
  ├─ [条件] enableUrlContext
  │   └─ providerToolPlugin('urlContext', config)     // 模型原生 urlContext 注入
  │
  ├─ [条件] isSupportedToolUse || isPromptToolUse
  │   └─ searchOrchestrationPlugin(assistant, topicId) // 搜索编排（意图分析 + 动态工具注入）
  │
  └─ [条件] isPromptToolUse
      └─ createPromptToolUsePlugin(mcpMode)           // Prompt 模拟工具调用
```

**注意**：`wrapLanguageModel` 会对 middleware 数组执行 `.reverse()`，因此数组中靠前的插件在运行时变成最外层包装。这也是 `ReasoningExtractionPlugin` 必须在 `SimulateStreamingPlugin` 之前推入的原因 — 反转后 `extractReasoning` 在外层，其状态机能处理 `simulateStreaming` 生成的模拟流中未闭合的标签。

## 插件详解

### TelemetryPlugin

- **触发条件**：developer mode 开启 + 存在 `topicId`
- **职责**：创建 OpenTelemetry span，通过 `AiSdkSpanAdapter` 将 AI SDK 的 span 转换为 Cherry Studio 格式
- **钩子**：`configureContext`（注入 tracer）、`onRequestStart`（创建 span）、`onRequestEnd` / `onError`（结束 span）

### PdfCompatibilityPlugin

- **触发条件**：始终启用（但跳过 OpenAI / Anthropic / Gemini 模型）
- **职责**：将 PDF `FilePart` 转换为 `TextPart`，使不支持原生 PDF 的 provider 仍能处理 PDF 附件
- **重要性**：必须在 Anthropic cache 插件之前执行，否则 cache 插件的 token 估算会基于错误的 FilePart 格式

### ReasoningExtractionPlugin

- **触发条件**：`provider === openai || azure-openai`
- **职责**：从响应中提取 `<thinking>`、`<｜begin▁of▁sentence｜>`、`<thought>` 等标签包裹的推理内容
- **标签名**：通过 `getReasoningTagName(model)` 根据模型动态确定标签名

### SimulateStreamingPlugin

- **触发条件**：`config.streamOutput === false`
- **职责**：使用 AI SDK 的 `simulateStreamingMiddleware` 将非流式响应模拟为流式输出
- **场景**：某些 provider 不支持原生流式输出时使用

### AnthropicCachePlugin

- **触发条件**：`provider.anthropicCacheControl.tokenThreshold` 存在
- **职责**：估算 token 数量，在 system message 和最后 N 条消息上添加 `cacheControl: { type: 'ephemeral' }`
- **效果**：利用 Anthropic 的 prompt caching 能力减少重复 token 的计费

### OpenrouterReasoningPlugin

- **触发条件**：`provider === openrouter`
- **职责**：从响应中移除 `[REDACTED]` 推理内容块（OpenRouter 对推理内容的脱敏处理）

### NoThinkPlugin

- **触发条件**：`provider === ovms` 且存在 MCP 工具
- **职责**：在用户消息后追加 `/no_think` 后缀，强制模型关闭思考模式
- **场景**：OVMS（OpenVINO 本地模型服务）+ MCP 工具场景下，思考模式会干扰工具调用

### QwenThinkingPlugin

- **触发条件**：Qwen3 系列模型 && provider 不支持 `enable_thinking`（排除 Ollama）
- **职责**：根据 `assistant.settings.reasoning_effort` 是否存在，在模型 ID 后追加 `/think` 或 `/no_think` 后缀
- **模型检测**：`isSupportedThinkingTokenQwenModel` && `!isQwen35to39Model`

### SkipGeminiThoughtSignaturePlugin

- **触发条件**：`isGemini3Model(model)`
- **职责**：在工具调用部分跳过 `thought_signature` 字段，避免 OpenAI 兼容 API 下 Gemini 3 模型的签名干扰

### SearchOrchestrationPlugin

最复杂的插件，负责三件事：

1. **`onRequestStart`**：发起一次轻量 `generateText()` 调用进行意图分析，判断是否需要网页搜索/知识搜索/记忆搜索
2. **`transformParams`**：动态注入对应工具（`KnowledgeSearchTool`、`MemorySearchTool`、`WebSearchTool`）
3. **`onRequestEnd`**：对话结束后异步触发记忆写回流程（不阻塞主回答）

**意图分析细节**：
- 使用 XML 结构返回识别结果（而非 schema object 生成）
- 识别失败时回退到用户原始输入作为保底 query
- 搜索触发不仅看开关，还综合 `assistant.webSearchProviderId`、知识库绑定、`knowledgeRecognition` 配置、全局 memory 配置

### PromptToolUsePlugin

- **触发条件**：`config.isPromptToolUse`（模型不支持原生 function calling）
- **职责**：通过 XML 协议模拟工具调用

**完整流程**：
1. `transformParams`：在 system prompt 中注入工具定义和使用示例
2. `transformStream`：创建 `TransformStream` 处理响应流
   - 使用 `TagExtractor` 过滤 `<tool_use>` 和 `<tool_use_result>` 标签，使其不出现在可见文本中
   - 在 `finish-step` 事件时检测是否有工具调用
   - 若有：通过 `ToolExecutor` 执行工具
   - 将工具结果格式化为 `<tool_use_result>` XML 注入流
   - 通过 `StreamEventManager` 管理递归调用和 usage 累积
3. 递归发起新请求，携带工具结果作为上下文

### providerToolPlugin

- **职责**：通用的 provider 原生能力注入插件
- **工作原理**：
  1. 接收 `ToolCapability`（`webSearch` 或 `urlContext`）和对应配置
  2. 通过 `extensionRegistry.resolveToolCapability()` 解析对应 provider 的工具工厂
  3. 支持 aggregator fallback（解析 `model.provider` 中的代理 provider 段，如 `aihubmix.google`）
  4. 将 `tools` 和 `providerOptions` 合并到请求参数中

## 工具调用的两条路径

### 路径 A：原生 Function Calling

适用于支持原生工具 schema 的模型（如 Claude、GPT-4o、Gemini Pro 等）。

```
AI SDK streamText({ tools: {...} })
  │
  ▼
模型返回 tool-call 事件
  │
  ▼
AiSdkToChunkAdapter 收到 tool-call 事件
  │
  ▼
委托 ToolCallChunkHandler 处理
  │
  ├─ 流式参数: tool-input-start → tool-input-delta → tool-input-end
  └─ 非流式: tool-call（一次性返回参数）
  │
  ▼
判断工具类型: builtin / MCP / provider
  │
  ▼
发送 MCP_TOOL_STREAMING / MCP_TOOL_PENDING / MCP_TOOL_COMPLETE 事件
  │
  ▼
UI 渲染工具调用状态
```

### 路径 B：Prompt Tool Use 模拟

适用于不支持原生 function calling 的模型。

```
PromptToolUsePlugin.transformParams()
  │
  ▼
在 system prompt 中注入工具定义 + 使用示例（XML 格式）
  │
  ▼
模型返回包含 <tool_use> 标签的文本
  │
  ▼
transformStream 中的 TagExtractor 流式提取工具调用
  │
  ▼
ToolExecutor.executeTools() 执行工具
  │
  ▼
StreamEventManager 发送 tool-call / tool-result 事件到流
  │
  ▼
格式化结果为 <tool_use_result> XML，注入流的下一轮
  │
  ▼
递归调用，将工具结果作为上下文发送给模型
  │
  ▼
AI SDK 递归深度控制（默认 max 10）
```

**两条路径在 UI 层统一**：无论走哪条路径，最终都转换为 `ChunkType` 事件，UI 渲染层无需感知差异。

### 两种路径的递归调用对比

原生 Function Calling 和 Prompt Tool Use 最大的区别在于"谁来控制多轮工具调用循环"。

**路径 A（原生）的递归在 AI SDK 内部**：

```typescript
// pluginEngine.ts → executeStreamWithPlugins()
// 第 381 行：executor 直接调 AI SDK 原生函数
const result = executor(resolvedModel, transformedParams, streamTransforms)
// → 展开为：
_streamText({ model: resolvedModel, tools: { ... }, ...transformedParams })
```

Cherry Studio 只负责把 `tools` 放进参数并调用 `_streamText()`。剩下的全自动：

```
_streamText({ tools: { ... } })
  │
  ▼ AI SDK 内部自动循环：
  │
  1. 发送 HTTP 请求给模型
  2. 模型返回 tool-call
  3. AI SDK 调用 tools[toolName].execute(input)
     → Cherry Studio 注册的 MCP 工具 / builtin 工具
  4. AI SDK 把 tool-result 回填给模型
  5. 模型基于工具结果继续生成文本
  6. 模型可能再次调用工具 → 回到第 2 步
  7. 直到模型不再调用工具，正常结束
```

AI SDK 内部通过 `maxSteps` 控制递归深度。Cherry Studio 侧通过 `stopWhen: stepCountIs(N)` 传入 `maxToolCalls` 配置来限制最大工具调用轮数。

**路径 B（Prompt 模拟）的递归在 Cherry Studio 代码里**：

```typescript
// pluginEngine.ts → executeStreamWithPlugins()
context.recursiveCall = async <R>(newParams) => {
  if (context.recursiveDepth >= context.maxRecursiveDepth) {
    throw new RecursiveDepthError(context.requestId, context.recursiveDepth, context.maxRecursiveDepth)
  }
  // 重新走一遍完整的插件生命周期
  return this.executeStreamWithPlugins(methodName, { ...params, ...newParams }, executor, context)
}
```

这个 `recursiveCall` 由 `PromptToolUsePlugin` 在检测到 XML 工具调用后触发，**重新发起一轮完整的 `executeStreamWithPlugins()`**：

```
PromptToolUsePlugin 的 TransformStream 拦截流
  │
  ├─ TagExtractor 检测到 <tool_use> 标签
  ├─ ToolExecutor.executeTools() 执行工具
  ├─ StreamEventManager 发送 tool-call/tool-result 事件
  ├─ 工具结果拼成 <tool_use_result> XML
  └─ context.recursiveCall({ messages: [...原消息, 工具结果] })
      │
      ▼
     重新走一遍 executeStreamWithPlugins()
      → resolveModel → transformParams → _streamText() → ...
```

深度由 `context.maxRecursiveDepth` 控制（默认 10）。

| | 原生 Function Calling | Prompt Tool Use 模拟 |
|---|---|---|
| **递归在哪** | AI SDK 内部（`_streamText()` 内） | Cherry Studio `PromptToolUsePlugin` |
| **触发方式** | AI SDK 自动检测模型返回的 tool-call | XML 标签解析后手动调 `recursiveCall()` |
| **谁执行工具** | AI SDK 调 `tools[name].execute()` | `ToolExecutor.executeTools()` |
| **谁回填结果** | AI SDK 自动构造下一轮消息 | 拼 `<tool_use_result>` XML 注入流 |
| **深度控制** | AI SDK 内部 `maxSteps` + Cherry Studio `stepCountIs(N)` | `context.maxRecursiveDepth`（默认 10） |
| **代码在哪** | `node_modules/ai` | `packages/aiCore/core/plugins/built-in/toolUsePlugin/` |
| **是否重新走插件生命周期** | 否（AI SDK 内部循环，不走 PluginEngine） | 是（每轮递归都重新执行 configureContext/transformParams 等） |

## ToolExecutor — 工具执行器

`ToolExecutor` 负责工具的执行、结果格式化和事件发送：

1. **executeTools()**：逐个执行工具，每个工具：
   - 发送 `tool-call` 事件到流
   - 调用 `tool.execute(input, { toolCallId, messages, abortSignal })`
   - 发送 `tool-result` 事件到流
   - 失败时发送 `tool-error` 事件

2. **formatToolResults()**：将执行结果格式化为 XML：
   ```xml
   <tool_use_result>
     <name>tool_name</name>
     <result>{"key": "value"}</result>
   </tool_use_result>
   ```

## StreamEventManager — 流事件管理

管理 Prompt Tool Use 模式下的递归流事件：

- **sendStepStartEvent()**：发送 `start-step` 事件
- **sendStepFinishEvent()**：发送 `finish-step` 事件，包含 usage 统计
- **Usage 累积**：支持 `LanguageModelUsage`、`ImageModelUsage`、`EmbeddingModelUsage` 三种类型的识别和跨递归调用累积
- **ProviderMetadata 合并**：合并各轮调用的 provider metadata
- **Messages 更新**：更新消息历史，确保递归调用的上下文正确

## TagExtractor — 流式 XML 标签提取器

通用流式 XML 标签提取器，处理跨 chunk 的不完整标签：

- 缓冲部分数据，检测 `<tag>` 和 `</tag>` 边界
- 正确处理标签跨多个 stream chunk 的情况
- 用于 `<tool_use>` 标签的流式解析

## 工具调用不只覆盖 MCP

"工具调用"在 Cherry Studio 中包含多种类型：

| 工具类型 | 来源 | 注入方式 |
|---------|------|---------|
| MCP 工具 | 主进程 MCPService | `setupToolsConfig()` → `tools` 参数 |
| 知识库搜索 | 渲染侧 `KnowledgeSearchTool` | 搜索编排插件动态注入 |
| 记忆搜索 | 渲染侧 `MemorySearchTool` | 搜索编排插件动态注入 |
| 网页搜索 | 渲染侧 `WebSearchTool` | 搜索编排插件 / providerToolPlugin |
| Provider 原生工具 | 各 provider 的 toolFactory | providerToolPlugin('webSearch' / 'urlContext') |

**模型原生 web search** 和 **外部 web search provider** 是两套不同的能力：
- 模型原生：通过 `providerToolPlugin('webSearch', config)` 注入，由模型内部执行搜索
- 外部 provider：通过 `searchOrchestrationPlugin` 注入 `WebSearchTool`，由 Cherry Studio 调用搜索 API 后把结果给模型

## 与 StreamProcessingService 的衔接

工具调用事件统一进入 `StreamProcessingService`，再映射到 UI 回调：

| 事件 | 回调 |
|------|------|
| 工具调用待执行 | `onToolCallPending` |
| 工具调用进行中 | `onToolCallInProgress` |
| 工具调用完成 | `onToolCallComplete` |
| 工具参数流式传输 | `onToolArgumentStreaming` |
| 外部工具执行中 | `onExternalToolInProgress` |
| 外部工具完成 | `onExternalToolComplete` |

这保证 UI 层无需感知底层工具协议差异，统一按事件类型渲染对应的 Block 组件。
