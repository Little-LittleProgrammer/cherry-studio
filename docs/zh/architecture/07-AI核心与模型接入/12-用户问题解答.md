## 一、Hub MCP 系统

### 1. Hub MCP 是干什么用的？

**A:** Hub MCP 是 Cherry Studio 的一个**内置元服务器（meta-server）**，用于**聚合所有已激活的 MCP Server**，并通过一组通用的元工具暴露给 LLM。

它是一个 "MCP 的 MCP"——把用户配置的所有 MCP Server 的工具统一管理，让 LLM 可以**动态发现和调用任意工具**，而不需要为每个 MCP Server 单独配置。

### 2. 暴露的 4 个元工具

| 工具 | 用途 |
| --- | --- |
| `list` | 分页列出所有可用的 MCP 工具 |
| `inspect` | 查看某个工具的参数签名（JSDoc 格式） |
| `invoke` | 调用单个工具 |
| `exec` | 执行 JavaScript 代码，编排多步工具调用 |

### 3. Auto Mode 工作流程

当助手设置为 **Auto 模式**时：

1. Hub Server 作为唯一的 MCP Server 注入
2. LLM 通过 `list` 发现工具 → `inspect` 了解参数 → `invoke`/`exec` 执行
3. 无需手动配置每个 MCP 工具，LLM 自主决定使用哪些能力

核心价值：

- **灵活性**：新增/删除 MCP Server 后无需重新配置助手，Hub 自动聚合
- **动态发现**：LLM 按需探索工具，而不是静态注入所有工具描述
- **Token 节省**：system prompt 中只带 4 个元工具的 schema，而非几十个真实工具的完整描述

### 4. Hub 如何调用其他 MCP 的？

Hub 本身不直接连接外部 MCP Server，而是通过 `mcp-bridge.ts` 桥接到 `MCPService`，由 `MCPService` 路由到具体的 MCP Server 实例。

**完整调用链路：**

```text
LLM 请求
  │
  ├─ invoke 模式 ────────────────────────────────────────────┐
  │   HubServer.handleInvoke()                                │
  │   → callMcpTool(name, params)                             │
  │                                                          │
  ├─ exec 模式                                               │
  │   Runtime.execute(code)                                   │
  │     └─ new Worker(hubWorkerSource)  // 独立 Worker 沙箱    │
  │          用户代码: mcp.callTool("xxx")                    │
  │          worker.postMessage("callTool")                   │
  │          ↓                                                │
  │     handleMessage → handleToolCall                        │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
                        │
                        ▼
              callMcpTool(nameOrId, params, callId)
              [mcp-bridge.ts:87]
                        │
              1. resolveToolId(nameOrId)
                 "githubSearchRepos" → "github__search_repos"
                        │
              2. mcpService.callToolById(toolId, params, callId)
                 [MCPService.ts:207]
                        │
              3. toolId.split('__')
                 → serverId = "github", toolName = "search_repos"
                        │
              4. 找到对应的 MCP Server 实例
                 server.client.callTool({ name, params })
                        │
                        ▼
              具体的 MCP Server (stdio/SSE/HTTP...)
```

**invoke vs exec 对比：**

| 维度 | `invoke` | `exec` |
| --- | --- | --- |
| 执行环境 | 主进程直接调用 | 独立 Node.js Worker（线程隔离） |
| 能力 | 调用单个工具 | 编排多步调用，支持 `parallel()`、`settle()` |
| 超时 | 跟随模型请求 | 固定 60 秒（`EXECUTION_TIMEOUT`） |
| 日志 | MCP Server 自身日志 | `mcp.log()` / `console.*` 回传，最多 1000 条 |

**exec 模式的 Worker 通信流程：**

```text
Runtime.execute(code)
  → 创建 Worker，postMessage("exec", code)
  → Worker 运行代码
  → Worker 调用 mcp.callTool("xxx") → postMessage("callTool")
  → 主线程收到 "callTool" → callMcpTool() → 调用目标 MCP
  → 结果 postMessage("toolResult") 回传 Worker
  → Worker 继续执行，最终 postMessage("result")
  → Runtime finalize，返回 ExecOutput
```

### 5. Hub 模式为什么通过 system prompt 注入，而不是 MCP 的 tools 字段注入？

实际上**两者都在用**，各司其职。

```typescript
// parameterBuilder.ts:220-235
if (tools) {
  params.tools = tools  // ← MCP tools 字段注入（4 个元工具的 schema）
}

if (getEffectiveMcpMode(assistant) === 'auto') {
  systemPrompt = `${systemPrompt}\n\n${getHubModeSystemPrompt()}`  // ← system prompt 注入（使用指南）
}
```

**分工：**

| 维度 | MCP tools 字段 | System prompt |
| --- | --- | --- |
| 注入内容 | 4 个元工具的结构化 schema | 使用指南、示例、关键规则 |
| 作用 | 让 LLM 知道有哪些工具及参数格式 | 教 LLM 怎么用这些工具 |
| 缺失后果 | LLM 看不到工具列表 | LLM 不知道最佳实践流程 |

**为什么不能只靠 tools 字段：**

Hub 模式需要教会 LLM 一套**方法论**，包括使用流程（list → inspect → invoke）、exec 可用的全局变量（`mcp.callTool()`、`mcp.log()` 等）、关键约束（exec 必须显式 return）、示例代码。这些**行为指南和示例**无法塞进 JSON Schema 的 description 字段。

**总结：** tools 字段 = "有什么"，system prompt = "怎么用"。

## 二、搜索编排提示词

### 1. 搜索编排提示词是干什么的？

**A:** 作用是**将对话中的后续查询重写为独立的、可检索的搜索查询**。

### 2. 为什么需要它

用户在多轮对话中提出的问题往往依赖上下文（如"它是什么？"、"那家公司呢？"），直接拿去搜索效果很差。这个提示词让 LLM 充当**"问题重述器"**，把对话式查询改写为独立的搜索关键词。

**示例：**

```text
用户：法国的首都是什么？
AI：巴黎。
用户：那里的人口有多少？     ← 不能直接搜"那里的人口有多少"
```

搜索编排提示词会把"那里的人口有多少？"重写为 **"巴黎的人口有多少？"**。

### 3. 用在哪里

用在 `src/renderer/src/aiCore/plugins/searchOrchestrationPlugin.ts`——**搜索编排插件**。

### 4. 完整工作流程

```text
用户发消息 → searchOrchestrationPlugin 拦截
  │
  ├─ 分析是否需要网络搜索 / 知识库检索
  │
  ├─ 选择合适的提示词：
  │    - 仅需网络搜索 → SEARCH_SUMMARY_PROMPT_WEB_ONLY
  │    - 仅需知识库   → SEARCH_SUMMARY_PROMPT_KNOWLEDGE_ONLY
  │    - 两者都需要   → SEARCH_SUMMARY_PROMPT（完整版）
  │
  ├─ 调用 LLM 重述问题（传入对话历史 + 当前问题）
  │
  └─ 解析输出：
       <websearch>
         <question>关键词</question>
         <links>https://...</links>
       </websearch>
       <knowledge>
         <rewrite>重写后的查询</rewrite>
         <question>原始查询</question>
       </knowledge>
  │
  └─ 根据解析结果触发网络搜索 / 知识库检索
```

### 5. 三个提示词变体

| 变体 | 使用场景 | 输出格式 |
| --- | --- | --- |
| `SEARCH_SUMMARY_PROMPT` | 网络搜索 + 知识库都需要 | `<websearch>` + `<knowledge>` |
| `SEARCH_SUMMARY_PROMPT_WEB_ONLY` | 仅网络搜索 | 仅 `<websearch>` |
| `SEARCH_SUMMARY_PROMPT_KNOWLEDGE_ONLY` | 仅知识库 | 仅 `<knowledge>`（含 `rewrite` 字段） |

### 6. 关键文件

| 文件 | 职责 |
| --- | --- |
| `src/renderer/src/config/prompts.ts` | 三个提示词变体定义 |
| `src/renderer/src/aiCore/plugins/searchOrchestrationPlugin.ts` | 搜索编排插件 |

## 三、aiCore 的 Plugin 与 Middleware

### 1. aiCore 中有两套增强机制？

**A:** 是的，aiCore 存在两套增强机制，分别来自不同来源：

| | Plugin（aiCore 插件） | Middleware（AI SDK 中间件） |
|---|---|---|
| 来源 | Cherry Studio 自己设计 | Vercel AI SDK v5 原生 |
| 类型 | `AiPlugin` | `LanguageModelV3Middleware` |
| 管理器 | `PluginEngine` + `PluginManager` | `wrapLanguageModel()` |
| 操作层级 | 高层业务请求生命周期 | 底层模型调用层（`doGenerate`/`doStream`） |

### 2. 两者的调用顺序

```
PluginEngine.executeStreamWithPlugins()
  │
  ├─ 0. configureContext           ← Plugin 可往 context.middlewares 注入 Middleware
  ├─ 1. onRequestStart             ← Plugin 钩子
  ├─ 2. resolveModel               ← Plugin 解析模型
  │
  ├─ 2.5 wrapLanguageModel(...)    ← ⚡ AI SDK Middleware 在这层生效
  │     (将 context.middlewares 包装到模型实例上)
  │
  ├─ 3. transformParams            ← Plugin 转换参数
  ├─ 4. collectStreamTransforms    ← Plugin 转换流
  ├─ 5. executor(...)              ← 实际 AI SDK 调用（经过 Middleware 拦截）
  ├─ 6. transformResult            ← Plugin 转换结果
  └─ 7. onRequestEnd               ← Plugin 钩子
```

### 3. 各自做什么

**Plugin 的能力：**

| 钩子 | 用途 |
|---|---|
| `resolveModel` | 模型解析（字符串 → LanguageModel 实例） |
| `configureContext` | 配置上下文，可注入 Middleware |
| `transformParams` | 转换高层请求参数 |
| `transformStream` | 返回 `TransformStream` 转换 AI 流 |
| `transformResult` | 转换返回结果 |
| `onRequestStart/End/Error` | 生命周期事件（并行执行） |

**Middleware 的能力：**

| 钩子 | 用途 |
|---|---|
| `transformParams` | 转换 AI SDK 原生参数（`LanguageModelV3CallOptions`） |
| `wrapGenerate` | 包装整个非流式生成操作 |
| `wrapStream` | 包装整个流式生成操作 |
| `overrideProvider/ModelId` | 覆写 provider/model 元信息 |

### 4. 同样的功能两套都能实现，会冲突吗？

**不会冲突**，因为它们在调用链的不同层级生效：

- **Plugin** 的 `transformParams` 操作的是 Cherry Studio 自己的高层参数（`GenerateTextParams`）
- **Middleware** 的 `transformParams` 操作的是 AI SDK 原生的低层参数（`LanguageModelV3CallOptions`）

两者是**协作关系**而非竞争关系。典型模式是 Plugin 在 `configureContext` 阶段往 `context.middlewares` 推送 Middleware，由 `PluginEngine` 统一通过 `wrapLanguageModel()` 应用。

### 5. 如何选择

| 场景 | 推荐方式 |
|---|---|
| 需要访问业务上下文（providerId、metadata、recursiveCall） | Plugin |
| 需要访问 AI SDK 原生参数（tool calls、providerOptions） | Middleware |
| 模型解析、模板加载、生命周期管理 | Plugin |
| 底层参数微调（cache control、thinking 模式后缀） | Middleware |

**现有示例：**

| 插件 | 方式 | 做了什么 |
|---|---|---|
| `anthropicCachePlugin` | Plugin → 注入 Middleware | 在 `transformParams` 中为系统消息添加 Anthropic cache control |
| `noThinkPlugin` | Plugin → 注入 Middleware | 在 `transformParams` 中为用户消息追加 `/no_think` 后缀 |
| `qwenThinkingPlugin` | Plugin → 注入 Middleware | 根据 `enableThinking` 追加 `/think` 或 `/no_think` |
| `searchOrchestrationPlugin` | 纯 Plugin | 拦截请求，调用 LLM 重述搜索查询 |

### 6. 关键文件

| 文件 | 职责 |
| --- | --- |
| `packages/aiCore/src/core/runtime/pluginEngine.ts` | PluginEngine 主类，编排 Plugin + 应用 Middleware |
| `packages/aiCore/src/core/plugins/types.ts` | `AiPlugin`、`AiRequestContext` 类型定义 |
| `packages/aiCore/src/core/plugins/manager.ts` | PluginManager，执行插件钩子 |
| `packages/aiCore/src/core/plugins/index.ts` | `createContext`、`definePlugin` 导出 |
| `src/renderer/src/aiCore/plugins/` | 各业务插件实现 |

## 四、Claude Code 集成

### 1. `spawnClaudeCodeProcess` 是干什么的？

**A:** 自定义 Claude Agent SDK CLI 子进程的启动方式。SDK 默认用 `child_process.spawn` 启动 CLI，这里通过 `Options.spawnClaudeCodeProcess` hook 替换为 `child_process.fork`。

**核心原因：Electron 环境下的代理注入。**

Electron 有自己的代理设置（系统代理/自定义代理），SDK 默认启动方式不会带上这些。自定义后能：

1. 检测当前代理配置
2. 通过 `--require` 注入代理 bootstrap 脚本，确保 SDK CLI 子进程正确走代理
3. 用 `fork` 替代 `spawn`，自带 IPC 通道，方便后续扩展

```typescript
// index.ts:515-557
spawnClaudeCodeProcess: (spawnOptions) => {
  const childEnv = { ...spawnOptions.env }
  let execArgv = process.execArgv

  const activeProxyConfig = getNodeProxyConfigFromEnvironment(childEnv)
  if (activeProxyConfig) {
    execArgv = [
      ...process.execArgv,
      '--disable-warning=UNDICI-EHPA',
      '--require',
      this.claudeProxyBootstrapPath  // 注入代理 bootstrap
    ]
  }

  const child = fork(spawnOptions.args[0], spawnOptions.args.slice(1), {
    cwd: spawnOptions.cwd,
    env: childEnv,
    execArgv,
    stdio: ['pipe', 'pipe', 'pipe', 'ipc'],
    signal: spawnOptions.signal
  })

  child.stderr?.on('data', (data) => { errorChunks.push(data.toString()) })
  return child as unknown as SpawnedProcess
}
```

### 2. 为什么用 `fork` 而非 `spawn`？有什么好处？

`fork` 是 `spawn` 的**特化版本**，专门用于启动 Node.js 子进程。

**对比：**

| | `spawn('node', 'cli.js')` | `fork('cli.js')` |
|---|---|---|
| 进程类型 | 任意可执行文件 | 仅 Node.js |
| IPC 通道 | 需手动配置 | **自带**（`stdio` 中加 `'ipc'`） |
| 通信方式 | 只能读写 stdin/stdout | **`child.send()` / `process.on('message')`** |

**好处：**

| 好处 | 说明 |
|---|---|
| 自带 IPC | `stdio: [..., 'ipc']` 建立双向通道，主进程可 `child.send()` 发消息，子进程可 `process.on('message')` 回传 |
| 代理注入 | 配合 `execArgv` 方便加 `--require` 等启动参数 |
| stderr 控制 | 保留管道访问，单独捕获 stderr 到日志 |
| 预留扩展 | 后续可在运行中动态注入配置、传递取消信号等 |

**现状：** 当前代码配了 `'ipc'` 但还没实际用上 `child.send()`/`child.on('message')`，主要是为后续预留双向通信能力。

### 3. Claude Code 整体在干什么？

UI 侧的 Agent 聊天不直接拼 Anthropic HTTP，而是由**主进程**调用 `@anthropic-ai/claude-agent-sdk` 的 `query()`。SDK 根据 `pathToClaudeCodeExecutable` 拉起内置 **CLI 子进程**，在 `cwd` 下执行「多轮对话 + 读写文件/Bash/工具」等 Claude Code 行为；模型 endpoint、密钥等通过**环境变量**注入子进程。

```text
invoke() 同步创建 ClaudeCodeStream 并 return
       → 调用方立即 stream.on('data', handler)   // 必须先订阅
       → setImmediate → processSDKQuery()
       → for await (query({ prompt: userInputStream, options }))
       → 每条 SDKMessage → transform → emit('data', { chunk })
       → 循环正常结束 → emit('data', { type: 'complete' })
```

### 4. `rtkRewrite` 是干什么的？

**A:** 对 Claude Code Agent 执行的 Bash 命令进行 **Token 优化重写**。

#### 做什么

Agent 决定执行 Bash 命令前，先把命令丢给一个捆绑的 CLI 工具 `rtk`，让它返回一个**等价但 Token 更少/输出更精简**的版本。

```typescript
// rtk.ts:151-171
export async function rtkRewrite(command: string): Promise<string | null> {
  const { stdout } = await execFileAsync(rtkPath, ['rewrite', command], { timeout: 3000 })
  const rewritten = stdout.trim()
  if (!rewritten || rewritten === command) return null
  return rewritten
}
```

#### 在 Claude Code 链路中的位置

作为 `PreToolUse` Hook 注册在 Agent 工具执行前：

```text
Agent 决定执行 Bash
  → PreToolUse Hook 触发
    → rtkRewriteHook: rtk rewrite "ls -la /some/path" → 返回优化后的等价命令
    → 用改写后的命令执行
```

```typescript
// index.ts:431-460
hooks: {
  PreToolUse: [{ hooks: [rtkRewriteHook, preToolUseHook] }]
}
```

**时机：** Agent 决定执行 Bash → `PreToolUse` Hook 触发 → `rtk` 重写命令 → 用改写后的命令执行

#### 目的

Claude Code Agent 会频繁执行 shell 命令（如 `ls`、`cat`、`grep` 等），这些命令的输出可能被截断或包含大量无用信息。`rtk` 能把命令改写成 **Token 更友好、输出更精简** 的等价形式，节省上下文窗口。

#### 来源

`rtk` 是随应用打包的 **Rust 二进制文件**（位于 `resources/binaries/{platform}/rtk`），启动时自动解压到 `~/.cherrystudio/bin/`：

- 仅支持特定平台（`win32-arm64` 不支持）
- 最低版本要求 `0.23.0`
- 超时 `3000ms`，超时或失败时静默回退原始命令
