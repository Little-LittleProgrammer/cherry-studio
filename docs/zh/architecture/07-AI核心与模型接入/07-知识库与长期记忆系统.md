# 07-知识库与长期记忆系统

知识库与记忆都属于“增强上下文”能力，但它们的目标和实现不同：

- 知识库：面向外部文档检索（RAG）
- 记忆：面向用户长期偏好与事实沉淀

## 知识库链路（RAG）

主进程入口：`src/main/services/KnowledgeService.ts`

核心实现要点：

1. 使用 `@cherrystudio/embedjs` 构建 RAG 应用。
2. 存储层基于 `LibSqlDb`，每个知识库独立路径。
3. 支持多源数据：文件、目录、网页、站点地图、笔记。
4. 通过 Embedding + 相似度搜索返回候选片段。
5. 可选 Reranker 做二次排序。

当前知识库链路还包含一些文档里未展开的重要能力：

- `RAGApplication` 会按知识库缓存复用，而不是每次查询临时创建。
- 启动时会清理上次异常退出遗留的待删除知识库目录。
- Loader 任务有 workload 和并发上限控制，避免大批量导入时拖垮主进程。
- 文档接入不只是原始 loader，还可经过预处理 provider。

渲染侧接入：

- 通过 `window.api.knowledgeBase.*` IPC 调用主进程服务。
- 在消息请求前，按策略注入知识搜索 prompt 或工具调用。

### 文档预处理与 Reranker

知识库并不只是“切 chunk + embedding”：

- PDF / 文档类内容可以走预处理 provider，再进入 RAG 导入流程。
- 当前预处理 provider 包括 `Doc2x`、`Mistral`、`MinerU`、`OpenMineru`、`PaddleOCR`、默认 provider。
- 检索后的候选结果可以再经过 `Reranker` 做二次排序。

因此，知识库链路实际上由“文档预处理 -> loader 导入 -> embedding 检索 -> rerank”四段组成。

## 记忆链路（Memory）

主进程入口：`src/main/services/memory/MemoryService.ts`

核心能力：

1. 本地 `libsql` 存储记忆与历史。
2. 记忆写入前做去重（hash + 相似度阈值）。
3. Embedding 向量检索（hybrid search）。
4. 维护记忆历史（ADD/UPDATE/DELETE 等语义）。

实现层补充：

- 启动时会执行旧版 memory DB 的迁移。
- 已删除但 hash 相同的记忆可被恢复，而不是重复插入新记录。
- 记忆向量当前会归一到统一维度，降低 embedding 模型切换带来的存量不兼容问题。
- vector index 创建失败时会降级到非索引搜索，而不是整体不可用。
- 检索和去重可结合 `userId` / `agentId` 做作用域隔离。

渲染侧入口：`src/renderer/src/services/MemoryService.ts`

职责：

- 维护当前用户上下文
- 调主进程 API 完成增删改查
- 同步记忆配置（embedding 模型、维度等）

## 搜索与记忆如何参与对话

`searchOrchestrationPlugin` 在一次请求中可触发：

1. 意图分析：判断是否需要知识/网页/记忆检索。
2. 动态注入工具：知识搜索、记忆搜索、网页搜索。
3. 对话结束后记忆写回：抽取事实并异步入库。

这样可以把“检索”与“写回”放入同一生命周期，而不是散落在页面逻辑。

补充：

- 知识检索是否触发不仅取决于绑定了知识库，还取决于 `knowledgeRecognition` 策略。
- 网页搜索与知识搜索可以同时参与同一轮请求。
- memory 写回不会阻塞用户当前消息返回。

## 设计边界

- 知识库关注“文档事实检索”，不存用户长期行为偏好。
- 记忆关注“用户长期状态”，不承担通用文档 RAG。
- 两者都由主进程持久化，渲染层只编排与展示。
