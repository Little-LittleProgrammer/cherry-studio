# 知识库与长期记忆系统

> 本文以**一次 LLM 请求的完整生命周期**为主线，串联知识库（RAG）和长期记忆系统的运作过程。

---

## 系统总览

Cherry Studio 在用户发送消息到 LLM 返回回复的过程中，会经历三个与知识/记忆相关的阶段：

```
用户发送消息
    │
    ▼
阶段一：知识库预检索 ────── 在请求发出前，先检索知识库，将相关文档注入提示词
    │
    ▼
阶段二：LLM 请求执行 ────── 发送请求，期间可动态调用知识/记忆搜索工具
    │
    ▼
阶段三：对话后记忆存储 ──── 请求结束后，从对话中提取事实存入长期记忆
```

三个子系统分别对应：

| 子系统           | 定位                 | 存储引擎          | 核心技术                                      |
| ---------------- | -------------------- | ----------------- | --------------------------------------------- |
| **知识库 (RAG)** | 文档级知识检索       | libSQL 向量数据库 | `@cherrystudio/embedjs` 向量嵌入 + 可选重排序 |
| **长期记忆**     | 对话事实提取与持久化 | libSQL 原生向量表 | LLM 事实提取 + 向量相似度去重                 |
| **知识图谱记忆** | 实体-关系-观测图谱   | JSON 文件         | MCP Memory Server（独立子系统）               |

---

## 入口：LLM 请求的起点

一切从 `ApiService.transformMessagesAndFetch()` 开始（`src/renderer/src/services/ApiService.ts`）。

这个函数是 LLM 请求的**总调度器**，它按顺序执行：

1. **消息预处理** — `ConversationService.prepareMessagesForModel()` 将内部消息格式转为 LLM 可理解的格式
2. **变量替换** — 替换提示词中的占位变量
3. **知识库预检索** — `injectUserMessageWithKnowledgeSearchPrompt()` ← **阶段一**
4. **发起 LLM 请求** — `fetchChatCompletion()` ← **阶段二**（内含阶段三）

## 阶段一：知识库预检索

### 1.1 触发入口

```
ApiService.transformMessagesAndFetch()
  └─> injectUserMessageWithKnowledgeSearchPrompt()    ← src/renderer/src/services/KnowledgeService.ts
        └─> getKnowledgeReferences()
              └─> processKnowledgeSearch()
```

在 LLM 请求发出**之前**，系统会同步检索当前助手关联的所有知识库，将检索结果以 `REFERENCE_PROMPT` 的形式注入到用户消息中。

### 1.2 检索流程详解

`processKnowledgeSearch()` 对每个关联的知识库执行以下步骤：

#### 步骤 1：构建 RAG 参数

```typescript
// src/renderer/src/services/KnowledgeService.ts
interface KnowledgeBaseParams {
  id: string; // 知识库 ID
  embeddingModel: Model; // 嵌入模型配置
  rerankModel?: Model; // 可选的重排序模型
  chunkSize: number; // 文档分块大小
  chunkOverlap: number; // 分块重叠量
  threshold: number; // 相似度阈值
  documentCount: number; // 最大返回文档数
  embedApiClient: any; // 嵌入模型的 API 客户端
  rerankApiClient?: any; // 重排序模型的 API 客户端
}
```

#### 步骤 2：IPC 调用主进程搜索

```
渲染进程                        主进程
  │                              │
  ├─ window.api.knowledgeBase    ├─ KnowledgeService.search()
  │   .search(params)            │   └─ ragApplication.search(query)
  │                              │       └─ LibSqlDb 向量数据库查询
  │                              │
  ├─ window.api.knowledgeBase    ├─ KnowledgeService.rerank() (可选)
  │   .rerank(params)            │   └─ GeneralReranker.rerank()
  │                              │       └─ HTTP 调用重排序模型 API
```

搜索过程：

1. 对用户查询文本做长度截断（适配嵌入模型的最大上下文）
2. 通过 IPC 调用主进程 `KnowledgeService.search()`
3. 主进程使用 `@cherrystudio/embedjs` 的 `RAGApplication` 执行向量检索
4. 按相似度阈值过滤结果
5. 如果配置了重排序模型，对结果进行二次排序
6. 限制返回数量为 `documentCount`

#### 步骤 3：生成引用块并注入提示词

```typescript
// src/renderer/src/services/KnowledgeService.ts
const references = await processKnowledgeSearch(params, query);
// 创建 UI 中显示的引用块
createCitationBlock(references);
// 将引用以 JSON 格式注入到用户消息中
const REFERENCE_PROMPT = `请基于以下参考资料回答问题：
${JSON.stringify(references)}

原始问题：${originalQuery}`;
```

最终用户的原始消息被改写为包含参考资料的增强消息。

### 1.3 工具调用路径：searchOrchestrationPlugin

除了预检索，知识库还可以通过 MCP 工具的方式被 LLM **主动调用**。这由 `searchOrchestrationPlugin` 控制：

```
src/renderer/src/aiCore/plugins/searchOrchestrationPlugin.ts

onRequestStart()
  └─ 使用 LLM 分析用户意图，提取知识搜索关键词

transformParams()
  └─ 将 knowledgeSearchTool 添加到 params.tools
       └─ builtin_knowledge_search 工具
            └─ KnowledgeSearchTool.execute()
                 └─ processKnowledgeSearch()  ← 与预检索使用同一套逻辑
```

这意味着 LLM 在生成回复的过程中，如果发现自己需要更多信息，可以**动态调用**知识库搜索工具。

### 1.4 主进程侧：RAG 引擎实现

```
src/main/services/KnowledgeService.ts

核心组件：
├─ RAGApplication          ← @cherrystudio/embedjs 的核心 RAG 应用构建器
├─ LibSqlDb                ← @cherrystudio/embedjs-libsql 向量存储后端
├─ Embeddings              ← 自定义嵌入封装（支持 OpenAI / Ollama / Voyage）
│   ├─ EmbeddingsFactory   ← 根据提供商创建对应的嵌入实现
│   └─ tracing             ← OpenTelemetry 链路追踪集成
├─ GeneralReranker         ← 通用重排序器（HTTP 调用外部重排序模型）
└─ PreprocessingService    ← PDF 预处理（doc2x / mistral / mineru / paddleocr）
```

每个知识库对应一个独立的 SQLite 数据库文件，存储在 `DataPath/KnowledgeBase/<sanitized-id>` 目录下。

### 1.5 知识库的数据类型与加载

知识库支持多种内容类型，每种类型对应不同的加载器：

| 类型         | 加载器            | 说明                                |
| ------------ | ----------------- | ----------------------------------- |
| 本地文件     | `LocalPathLoader` | PDF、DOCX、TXT 等，PDF 会经过预处理 |
| 网页 URL     | `WebLoader`       | 直接抓取网页内容                    |
| 站点地图     | `SitemapLoader`   | 批量加载整个站点                    |
| 笔记         | `NoteLoader`      | Cherry Studio 内部笔记              |
| 目录         | `LocalPathLoader` | 递归扫描目录下所有支持的文件        |
| EPUB         | `EpubLoader`      | 电子书格式                          |
| OpenDocument | `OdLoader`        | ODT/ODS/ODP 格式                    |

**索引流程**：

```
用户添加内容
  └─> Redux 标记为 processingStatus: 'pending'
       └─> KnowledgeQueue 轮询 pending 项
            └─> window.api.knowledgeBase.add() (IPC)
                 └─> KnowledgeService.add() (主进程)
                      └─> 根据文件类型选择加载器
                           └─> ragApplication.addLoader()
                                └─> 分块 → 嵌入 → 存入 LibSqlDb
```

## 阶段二：LLM 请求执行

### 2.1 请求构建

在知识库预检索完成后，进入 `fetchChatCompletion()`：

```typescript
// src/renderer/src/services/ApiService.ts
await fetchChatCompletion({
  messages: modelMessages, // 已注入知识库引用
  assistant,
  allowedTools, // 可能包含 builtin_knowledge_search, builtin_memory_search
  onChunkReceived,
});
```

### 2.2 可用的知识/记忆工具

如果启用了工具调用能力，以下内置工具会被添加到请求中：

| 工具名                     | 功能             | 实现文件                                               |
| -------------------------- | ---------------- | ------------------------------------------------------ |
| `builtin_knowledge_search` | 知识库语义搜索   | `src/renderer/src/aiCore/tools/KnowledgeSearchTool.ts` |
| `builtin_web_search`       | 网络搜索         | （Web 搜索模块）                                       |
| `builtin_memory_search`    | 长期记忆向量搜索 | `src/renderer/src/aiCore/tools/MemorySearchTool.ts`    |

### 2.3 记忆搜索工具的执行路径

当 LLM 决定调用 `builtin_memory_search` 时：

```
LLM 调用 builtin_memory_search 工具
  └─> MemorySearchTool.execute({ query })
       └─> MemoryProcessor.searchRelevantMemories(query)
            └─> MemoryService.search() (渲染进程)
                 └─> window.api.memory.search() (IPC)
                      └─> MemoryService.search() (主进程)
                           └─> 生成查询向量 → hybridSearch() (余弦相似度)
                                └─> 返回 Top-K 记忆列表
```

## 阶段三：对话后记忆存储

### 3.1 触发时机

LLM 请求结束后，`searchOrchestrationPlugin.onRequestEnd()` 被调用：

```typescript
// src/renderer/src/aiCore/plugins/searchOrchestrationPlugin.ts
onRequestEnd() {
  if (globalMemoryEnabled && assistant.enableMemory) {
    storeConversationMemory(messages, assistant)
  }
}
```

只有同时满足两个条件才会触发记忆存储：

- 全局记忆开关已启用 (`globalMemoryEnabled`)
- 当前助手启用了记忆 (`assistant.enableMemory`)

### 3.2 事实提取与存储流程

```
storeConversationMemory()
  └─> MemoryProcessor.processConversation()
       │
       ├─ 步骤1: extractFacts()
       │    └─ 将对话历史发送给 LLM
       │         └─ 使用 factExtractionPrompt 提示词
       │              └─ LLM 返回 { facts: string[] }
       │
       ├─ 步骤2: updateMemories()
       │    └─ 将已有记忆 + 新事实发送给 LLM
       │         └─ 使用 updateMemorySystemPrompt 提示词
       │              └─ LLM 返回 [{ id, text, event: 'ADD'|'UPDATE'|'DELETE'|'NONE' }]
       │
       └─ 步骤3: 执行记忆操作
            ├─ MemoryService.add() → 新增记忆
            ├─ MemoryService.update() → 更新记忆
            └─ MemoryService.delete() → 删除记忆
```

### 3.3 主进程：记忆的持久化

`MemoryService.add()` 的执行细节：

```
src/main/services/memory/MemoryService.ts

1. 去重检查
   └─ 计算 SHA-256 哈希，避免存储重复内容

2. 生成嵌入向量
   └─ 使用配置的嵌入模型将文本转为向量

3. 相似度去重
   └─ hybridSearch() 检查是否已有相似度 >= 0.85 的记忆
        └─ 如果有，跳过添加（避免重复记忆）

4. 插入数据库
   └─ INSERT INTO memories (id, memory, hash, embedding, metadata, ...)
        └─ 使用 libSQL 原生 F32_BLOB 向量类型

5. 记录操作历史
   └─ INSERT INTO memory_history (operation, memory_id, ...)
```

### 3.4 数据库结构

记忆系统使用独立的 libSQL/SQLite 数据库，路径为 `DataPath/Memory/memories.db`：

```sql
-- 记忆主表
CREATE TABLE memories (
  id TEXT PRIMARY KEY,
  memory TEXT NOT NULL,           -- 记忆内容
  hash TEXT UNIQUE,               -- SHA-256 去重哈希
  embedding F32_BLOB(1536),       -- libSQL 原生向量类型
  metadata TEXT,                  -- JSON 元数据
  user_id TEXT,                   -- 用户 ID
  agent_id TEXT,                  -- 助手 ID
  run_id TEXT,                    -- 会话运行 ID
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  is_deleted INTEGER DEFAULT 0
);

-- 操作历史表（审计追踪）
CREATE TABLE memory_history (
  id TEXT PRIMARY KEY,
  operation TEXT NOT NULL,        -- ADD / UPDATE / DELETE
  memory_id TEXT,
  old_memory TEXT,
  new_memory TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 向量索引（加速余弦相似度查询）
CREATE INDEX libsql_vector_idx ON memories (libsql_vector_idx(embedding));
```

### 3.5 记忆搜索的两种模式

记忆搜索支持两种模式：

| 模式         | 触发方式       | 说明                                                                    |
| ------------ | -------------- | ----------------------------------------------------------------------- |
| **向量搜索** | 配置了嵌入模型 | 将查询文本转为向量，使用 `vector_distance_cos` 计算余弦相似度，阈值 0.5 |
| **文本搜索** | 未配置嵌入模型 | 使用 SQL `LIKE` 进行模糊文本匹配                                        |

## 子系统：MCP 知识图谱记忆

除了上述向量记忆系统外，还有一个独立的**知识图谱**记忆系统：

```
src/main/mcpServers/memory.ts

使用实体-关系-观测（Entity-Relation-Observation）模型
  └─ 数据存储在 JSON 文件中
  └─ 作为 MCP Server 对外提供工具
       └─ LLM 可以通过 MCP 工具调用创建/查询/更新知识图谱
```

这个系统与向量记忆系统是**独立**的，提供了更结构化的知识表示能力。

## IPC 通信通道

渲染进程与主进程之间通过以下 IPC 通道进行通信：

### 知识库通道

| 通道                   | 功能             |
| ---------------------- | ---------------- |
| `KnowledgeBase_Create` | 创建知识库       |
| `KnowledgeBase_Reset`  | 重置知识库       |
| `KnowledgeBase_Delete` | 删除知识库       |
| `KnowledgeBase_Add`    | 添加文档到知识库 |
| `KnowledgeBase_Remove` | 从知识库移除文档 |
| `KnowledgeBase_Search` | 搜索知识库       |
| `KnowledgeBase_Rerank` | 重排序搜索结果   |

### 记忆通道

| 通道                              | 功能             |
| --------------------------------- | ---------------- |
| `Memory_Add`                      | 添加记忆         |
| `Memory_Search`                   | 搜索记忆         |
| `Memory_List`                     | 列出记忆         |
| `Memory_Delete`                   | 删除记忆         |
| `Memory_Update`                   | 更新记忆         |
| `Memory_Get`                      | 获取单条记忆     |
| `Memory_SetConfig`                | 设置记忆配置     |
| `Memory_DeleteUser`               | 删除用户记忆     |

## 状态管理

### Redux 存储

| Slice       | 存储内容                               | 文件                                  |
| ----------- | -------------------------------------- | ------------------------------------- |
| `knowledge` | 知识库列表、每个库的条目与处理状态     | `src/renderer/src/store/knowledge.ts` |
| `memory`    | 记忆配置（嵌入模型、LLM 模型、提示词） | `src/renderer/src/store/memory.ts`    |

### 记忆配置结构

```typescript
interface MemoryConfig {
  embeddingModel?: Model; // 嵌入模型
  embeddingDimensions?: number; // 嵌入维度
  llmModel?: Model; // 用于事实提取的 LLM
  customFactExtractionPrompt?: string; // 自定义事实提取提示词
  customUpdateMemoryPrompt?: string; // 自定义记忆更新提示词
}
```

每个助手也可以单独配置是否启用记忆：

```typescript
interface Assistant {
  enableMemory?: boolean; // 助手级别的记忆开关
}
```

## 总结：一次完整请求的数据流

```
用户发送 "如何在 Cherry Studio 中配置 MCP？"
│
├── 阶段一：知识库预检索
│   ├── 获取助手关联的知识库列表
│   ├── 对每个知识库执行向量搜索
│   │   ├── 渲染进程 IPC → 主进程 KnowledgeService.search()
│   │   ├── ragApplication.search() 查询 libSQL 向量数据库
│   │   └── 可选：调用重排序模型进行二次排序
│   ├── 创建引用块（UI 展示）
│   └── 将参考资料以 JSON 格式注入用户消息
│
├── 阶段二：LLM 请求
│   ├── 构建请求参数（含注入后的消息 + MCP 工具列表）
│   ├── 注册搜索编排插件（searchOrchestrationPlugin）
│   ├── 插件分析意图，动态添加知识/记忆搜索工具
│   ├── 流式发送请求，LLM 可能调用：
│   │   ├── builtin_knowledge_search → 再次检索知识库
│   │   └── builtin_memory_search → 检索长期记忆
│   └── 接收并渲染流式回复
│
└── 阶段三：记忆存储
    ├── 请求结束，触发 onRequestEnd
    ├── 检查全局+助手级记忆开关
    ├── MemoryProcessor.processConversation()
    │   ├── LLM 提取对话中的事实 → { facts: [...] }
    │   ├── LLM 对比已有记忆 → 决定 ADD/UPDATE/DELETE
    │   └── 执行对应操作
    └── 主进程 MemoryService.add()
        ├── 计算哈希去重
        ├── 生成嵌入向量
        ├── 检查相似度 >= 0.85 的已有记忆
        ├── 插入 libSQL 数据库
        └── 记录到 memory_history 审计表
```
