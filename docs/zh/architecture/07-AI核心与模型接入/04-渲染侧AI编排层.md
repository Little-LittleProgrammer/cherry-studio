# 04-渲染侧AI编排层

渲染侧 AI 编排层位于 `src/renderer/src/aiCore` 与 `src/renderer/src/services/ApiService.ts`，负责把产品语义转换为可执行请求。

## 入口结构

```
src/renderer/src/aiCore/
├── AiProvider.ts                    # 核心 Provider 类（统一入口）
├── index.ts                         # 导出 AiProvider
├── prepareParams/                   # 参数准备与消息转换
├── provider/                        # Provider ID 映射与适配
├── plugins/                         # 插件装配与实现
├── chunk/                           # AI SDK 流事件 -> Chunk 转换
├── tools/                           # 内置工具（知识搜索/记忆搜索/网页搜索）
├── services/                        # 模型列表获取
├── trace/                           # OpenTelemetry span 适配
├── types/                           # 类型定义
└── utils/                           # 工具函数（reasoning/options/image/websearch/mcp）
```

**注意**：不再存在 `index_new.ts` 或 `ModernAiProvider`。当前的统一入口是 `AiProvider` 类，位于 `AiProvider.ts`。

## AiProvider 的职责

`AiProvider` 是渲染侧唯一的 Provider 类，承担以下关键职责：

### 构造模式

支持三种构造函数签名：

```typescript
// 仅传入 Model，自动推导 provider
const ai = new AiProvider(model)

// 传入 Model + 自定义 Provider
const ai = new AiProvider(model, customProvider)

// 仅传入 Provider（用于 fetchModels 等不需要绑定 model 的场景）
const ai = new AiProvider(provider)
```

构造时自动执行 `adaptProvider()` 和 `providerToAiSdkConfig()`，将 Cherry Studio 的 Provider 配置转换为 AI SDK 所需的配置格式。

### completions() — 主聊天入口

`completions(modelId, params, middlewareConfig)` 是文本对话的核心方法：

1. 检查 developer mode + `topicId` → 进入 `_completionsForTrace()` 带 trace 的分支
2. 否则进入 `modernCompletions()` 标准分支

`modernCompletions()` 的执行步骤：

1. 调用 `buildPlugins()` 组装插件数组
2. 通过 `createExecutor(providerId, providerSettings, plugins)` 从 `@cherrystudio/ai-core` 创建执行器
3. 调用 `executor.streamText()` 发起流式请求
4. 若有 `onChunk` 回调：创建 `AiSdkToChunkAdapter` 并调用 `processStream()` 转换流事件
5. 若无 `onChunk`：强制消费流（`consumeStream()`）防止阻塞，返回 `getText()` + `usage`

**关键设计**：无论是否有 `onChunk`，都使用 `streamText`，不使用 `generateText`。无回调时通过 `consumeStream()` 消费流并保留真实错误诊断。

### Anthropic OAuth 特殊处理

当 `provider.id === 'anthropic'` 且 `authType === 'oauth'` 时，system prompt 会被转换为 Claude Code 兼容的系统消息结构：

```typescript
const claudeCodeSystemMessage = buildClaudeCodeSystemModelMessage(systemParam)
params.system = undefined
params.messages = [...claudeCodeSystemMessage, ...(params.messages || [])]
```

### 图像生成与编辑

- `generateImage(params)` — 图像生成，通过 `executor.generateImage()` 实现
- `editImage(params)` — 图像编辑，通过 `executor.generateImage()` 并传入 `prompt.images` 实现
- `getEmbeddingDimensions(model)` — 通过 `executor.embedMany()` 测试获取嵌入维度

### API Key 轮转

`getApiKey()` 方法处理多 key 轮转：
- 单 key 直接返回
- 多 key 通过 `window.keyv` 记录上次使用的 key，下次自动切换到下一个

## ApiService 的业务调度

`ApiService.ts` 是更上层的业务入口，负责：

### transformMessagesAndFetch()

聊天请求的最高层入口：

1. 调用 `ConversationService.prepareMessagesForModel()` 获取 `modelMessages` 和 `uiMessages`
2. 替换 prompt 中的变量
3. 检测专用图像生成模型 → 路由到 `fetchImageGeneration()`
4. 注入知识搜索 prompt
5. 委托给 `fetchChatCompletion()`

### fetchChatCompletion()

主聊天函数，完整流程：

1. 获取 provider（含 API key 轮转）
2. 创建 `AiProvider` 实例
3. 获取 MCP 工具列表（若启用工具调用）
4. 调用 `buildStreamTextParams()` 构建 AI SDK 参数
5. 构建 `AiSdkMiddlewareConfig` 能力标志
6. 包装 `onChunkReceived` 回调（自动追踪 token 使用量）
7. 调用 `AI.completions(modelId, aiSdkParams, middlewareConfig)`

### 其他辅助函数

- `fetchMessagesSummary()` — 消息摘要
- `fetchNoteSummary()` — 笔记摘要
- `fetchGenerate()` — 文本生成
- `fetchModels()` — 模型列表拉取
- `checkApi()` / `checkModel()` — API/模型连通性检查

## 参数准备层（prepareParams/）

`prepareParams/` 负责把对话与文件转换为 AI SDK 参数：

| 文件 | 职责 |
|------|------|
| `messageConverter.ts` | Cherry Studio Message -> AI SDK ModelMessage 转换，含视觉模型特殊处理（合并 assistant 图片到 user 消息）、base64 图片剥离（防 HTTP 413） |
| `fileProcessor.ts` | 文件内容提取，PDF 原生 FilePart 支持，Gemini File API / OpenAI file upload 大文件处理，文本提取降级 |
| `modelCapabilities.ts` | 模型能力检测：`supportsImageInput()`、`supportsLargeFileUpload()`、`getFileSizeLimit()`，白名单+黑名单模式 |
| `parameterBuilder.ts` | 参数总装配 `buildStreamTextParams()`：整合 IdleTimeoutController、MCP 工具配置、providerOptions、webSearch 配置、Anthropic beta headers、温度/topP/maxTokens、stepCountIs（最大工具调用次数）、MCP auto-mode system prompt 注入 |
| `modelParameters.ts` | 标量参数获取：`getTemperature()`、`getTopP()`、`getTimeout()`、`getMaxTokens()`，尊重模型能力与助手设置 |
| `header.ts` | Anthropic beta headers：Claude 4.5 + function tool use 的 `interleaved-thinking-2025-05-14`、Vertex Claude 4 + web search 的 `web-search-2025-03-05` |

### buildStreamTextParams() 详解

这是参数准备的核心函数，决定：

- **工具调用可用性**：通过 `setupToolsConfig()` 配置 MCP 工具
- **文件输入可用性**：通过 `fileProcessor` 处理文件/图片
- **模型专属参数**：温度、topP、maxTokens、timeout（IdleTimeoutController，支持 reset）
- **推理能力**：`enableReasoning` → 通过 `getReasoningEffort()` 映射到 20+ 模型家族的正确参数形状
- **Web 搜索**：`enableWebSearch` → 模型原生搜索 vs 外部 web search provider
- **URL 上下文**：`enableUrlContext` → Anthropic urlContext / Google urlContext
- **最大工具调用次数**：`stepCountIs` 设置，对应助手的 max tool calls 配置

## Provider 适配层（provider/）

`provider/factory.ts` 与 `provider/providerConfig.ts` 负责：

### Provider ID 映射

`getAiSdkProviderId()` 将 Cherry Studio Provider 映射到 AI SDK provider ID：
- Azure endpoint 检测 → `azure-anthropic` 或 `azure-openai`
- OpenAI API domain 检测 → 区分真实 OpenAI 与兼容层
- 别名解析 → 如 `tokenflux` → `openrouter`

### Provider 配置转换

`providerToAiSdkConfig()` 使用策略模式，为每种 provider 类型构建配置：
- Copilot、Ollama、Bedrock、Vertex、Azure、CherryAI、NewAPI、AiHubMix、generic、openai-compatible
- 每种类型有独立的 config builder

### API Host 格式化

`formatProviderApiHost()` 标准化 API URL：
- Anthropic → `https://api.anthropic.com`
- Ollama → 本地地址
- Gemini → `https://generativelanguage.googleapis.com`
- Azure → 添加 `/anthropic/v1` 路径后缀
- Vertex → 自动添加版本后缀

### Provider Extension 注册

在 `provider/extensions/index.ts` 中注册了 15+ provider 扩展到 `@cherrystudio/ai-core/provider` 的 `extensionRegistry`：

| 扩展 | 特殊能力 |
|------|----------|
| GoogleVertex | webSearch toolFactory |
| GoogleVertexAnthropic | Vertex 托管的 Claude |
| GitHubCopilot | 默认 headers |
| Bedrock | AWS 凭证 |
| Perplexity | - |
| Mistral | - |
| HuggingFace | - |
| Gateway | AI SDK gateway 模式 |
| Cerebras | - |
| Ollama | 本地部署 |
| AiHubMix | 代理 provider |
| NewAPI | 代理 provider |

## 插件编排层（plugins/）

`PluginBuilder.ts` 的 `buildPlugins()` 函数根据 provider/model/config 条件组装插件。完整插件列表见 `05-插件系统与工具调用.md`，这里仅列出与编排层的关系。

装配顺序（执行时按 `enforce` 排序）：

1. **TelemetryPlugin** — developer mode + topicId
2. **PdfCompatibilityPlugin** — 始终启用（跳过 OpenAI/Anthropic/Gemini）
3. **ReasoningExtractionPlugin** — OpenAI/Azure provider 的 `<thinking>`/`<｜begin▁of▁sentence｜>` 标签提取
4. **SimulateStreamingPlugin** — `streamOutput` 为 false 时
5. **AnthropicCachePlugin** — 有 `anthropicCacheControl.tokenThreshold` 时
6. **OpenrouterReasoningPlugin** — OpenRouter 的 `[REDACTED]` 清洗
7. **NoThinkPlugin** — OVMS + MCP 工具场景强制关闭思考
8. **QwenThinkingPlugin** — Qwen3 在无 `enable_thinking` 的 provider 上的 `/think`/`/no_think` 后缀
9. **SkipGeminiThoughtSignaturePlugin** — Gemini 3 模型的思考签名干扰跳过
10. **providerToolPlugin(webSearch)** — `enableWebSearch` + 有配置时
11. **providerToolPlugin(urlContext)** — `enableUrlContext` 时
12. **SearchOrchestrationPlugin** — 工具调用启用时的意图分析 + 动态搜索工具注入 + 记忆写回
13. **PromptToolUsePlugin** — prompt 模拟工具调用时

## Chunk 适配层（chunk/）

`chunk/AiSdkToChunkAdapter.ts` 把 AI SDK 流事件转换为 UI 可消费的 `ChunkType`：

| AI SDK 事件 | ChunkType |
|------------|-----------|
| `text-start` | `TEXT_START` |
| `text-delta` | `TEXT_DELTA` |
| `text-end` | `TEXT_COMPLETE` |
| `reasoning-start` | `THINKING_START` |
| `reasoning-delta` | `THINKING_DELTA` |
| `reasoning-end` | `THINKING_COMPLETE` |
| `tool-call` / `tool-input-*` / `tool-result` / `tool-error` | 委托 `ToolCallChunkHandler` |
| `finish` | `BLOCK_COMPLETE` + `LLM_RESPONSE_COMPLETE`（含 usage/metrics） |
| `source` | 累积 web search 结果 |
| `abort` / `error` | `ERROR` |

`ToolCallChunkHandler`（`handleToolCallChunk.ts`）处理工具调用的状态机：
- 全局静态 Map 跟踪活跃工具调用
- 流式参数（`tool-input-start/delta/end`）与非流式（`tool-call`）
- 区分工具类型：builtin、MCP、provider
- 发送 `MCP_TOOL_STREAMING`、`MCP_TOOL_PENDING`、`MCP_TOOL_COMPLETE` 事件
- 从工具输出中提取图片

### 额外职责

- 首 token 计时
- idle timeout reset（每次收到 chunk 重置超时计数器）
- link 转换（web search 启用时将 URL 转为可点击链接）

## 与 ApiService 的协作边界

`ApiService` 负责"业务调度"：
- 从 store 读取助手配置
- 计算 MCP 工具集合
- 注入知识检索 prompt
- 处理 web search provider、知识库、memory 等产品配置
- 管理中止控制器
- 发起模型拉取等 provider 级辅助调用
- 将 `Chunk` 回调给 UI 流处理器

`AiProvider` 负责"执行组织"：
- Provider 适配与初始化
- 插件装配
- Executor 创建与调用
- 流事件转换
- API key 轮转
- Trace span 管理

`ApiService` 持有业务上下文，`AiProvider` 持有执行上下文。两者协作但不互相污染。
