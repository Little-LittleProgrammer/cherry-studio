# 04-渲染侧AI编排层

渲染侧 AI 编排层位于 `src/renderer/src/aiCore` 与 `src/renderer/src/services/ApiService.ts`，负责把产品语义转换为可执行请求。

## 入口结构

- `src/renderer/src/aiCore/index.ts`：兼容入口（默认 legacy，导出 Modern）
- `src/renderer/src/aiCore/index_new.ts`：`ModernAiProvider` 主实现
- `src/renderer/src/services/ApiService.ts`：业务入口与请求编排
- `src/renderer/src/aiCore/prepareParams/`：参数转换
- `src/renderer/src/aiCore/provider/`：Provider 适配与初始化
- `src/renderer/src/aiCore/chunk/`：流事件适配

## ModernAiProvider 的职责

`index_new.ts` 中 `ModernAiProvider` 做四件关键事：

1. 识别调用上下文（文本、图像、Trace、是否回退 legacy）。
2. 将产品 Provider 配置转换为 aiCore provider config。
3. 延迟初始化本地 provider 实例（按需创建）。
4. 组装插件并调用 aiCore 执行器。

它是“产品侧策略开关”汇聚点，不是纯协议层。

除此之外，`ModernAiProvider` 还有几个容易遗漏的特殊分支：

- 支持 `(provider)` 构造模式，供 `fetchModels` 等无需绑定具体 model 的路径复用同一套 provider 适配逻辑。
- 遇到 dedicated image endpoint 时，会识别是否进入图像生成分支。
- 非 gateway 的图像生成端点，当前仍可能回退 legacy 图像链路，以支持图片编辑等历史能力。
- Anthropic OAuth 模式下会把 system prompt 转换为 Claude Code 兼容的系统消息结构。
- developer mode 且存在 `topicId` 时，会进入 trace 专用执行分支。

## 参数准备层

`prepareParams/` 负责把对话与文件转换为 AI SDK 参数：

- `messageConverter.ts`：消息结构转换
- `fileProcessor.ts`：文件内容与附件处理
- `modelCapabilities.ts`：模型能力检测
- `parameterBuilder.ts`：参数总装配（`buildStreamTextParams`）

这一层决定：

- 工具调用可用性
- 文件输入可用性
- 模型专属参数（温度、推理强度等）
- 请求级超时与 header

其中和“模型接入能力矩阵”强相关的细节包括：

- 视觉模型检测：是否支持原生图片输入。
- 大文件上传能力检测：如 Gemini File API、特定 Qwen 长文档模型。
- provider 特定文件大小阈值：例如 Anthropic PDF 限制、Gemini 小文件阈值、Dashscope 特例。
- reasoning / thinking / tool use / image endpoint 等能力最终会在这层折叠为执行参数。

## Provider 适配层

`provider/factory.ts` 与 `provider/providerConfig.ts` 负责：

1. Provider ID 映射（系统 provider、别名、兼容 provider）。
2. OpenAI/Azure/Hub 等特殊模式处理。
3. Provider 初始化（动态注册 + 本地实例化）。
4. 特殊 provider 配置预处理（例如 OAuth、兼容 API host、gateway 路径差异）。

核心目标是“统一入口 + 差异收敛”。

## Chunk 适配层

`chunk/AiSdkToChunkAdapter.ts` 把 AI SDK 流事件转换为 UI 可消费 `ChunkType`：

- 文本：`TEXT_START/DELTA/COMPLETE`
- 思考：`THINKING_START/DELTA/COMPLETE`
- 工具：`MCP_TOOL_PENDING/IN_PROGRESS/COMPLETE`
- 外部工具：知识库、网页搜索完成事件
- 图片：创建、增量、完成
- Agent 原始事件：`RAW`
- 错误/中止：统一错误语义

这一层消除了不同 provider 的流协议差异。

补充说明：

- Agent / Claude Code 流式原始消息也会经过这里，被转成统一 `Chunk` 事件。
- 适配器还负责处理 provider metadata、首 token 时间、idle timeout reset 等流式运行时信息。

## 与 ApiService 的协作边界

`ApiService` 负责“业务调度”：

- 从 store 读取助手配置
- 计算 MCP 工具集合
- 注入知识检索 prompt
- 处理 web search provider、知识库、memory 等产品配置
- 管理中止控制器
- 发起模型拉取等 provider 级辅助调用
- 将 `Chunk` 回调给 UI 流处理器

`ModernAiProvider` 负责“执行组织”，`ApiService` 负责“业务上下文”。
